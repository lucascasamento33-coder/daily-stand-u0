Parallel Tool Calls: When Sequential Beats Concurrent
You're probably defaulting to parallel tool calls whenever Claude needs multiple pieces of data at once, but there's a specific scenario where forcing sequential execution cuts your total latency in half. When one tool's output directly constrains or focuses the next tool's work—think fetching a tenant's lease ID before pulling their payment history, or grabbing a property's Buildium unit list before querying maintenance tickets—running them in parallel wastes tokens and time because the second call can't use the first's result to narrow its scope. The win isn't intuitive: sequential feels slower, but when the second tool can skip scanning irrelevant records or avoid a broad search because it already has the exact ID or filter criteria, you save more time on execution than you lose waiting. Here's the pattern: in your scheduled task that audits overdue maintenance tickets, instead of parallel-calling "get all properties" and "get all open tickets," chain them—fetch properties first, filter to the three with upcoming inspections, then call for tickets scoped to just those property IDs. That second call runs in a fraction of the time because Buildium's API isn't scanning your entire portfolio. The flip side: when tools are truly independent—checking weather, pulling rent comps, and fetching a PDF from PandaDocs—parallel wins every time. The decision point is dependency: if tool B's arguments or filters benefit from tool A's output, chain them. Your PreToolUse hook is the perfect place to enforce this: inspect the tool queue, and if you see a payment-history call queued alongside a lease-lookup call for the same tenant, delay the payment call until the lease ID is in context. For your rent-roll audits, this pattern alone can drop runtime from 40 seconds to under 20 when you're processing 50-plus units, because each downstream Buildium call is laser-focused instead of scanning the whole dataset.
